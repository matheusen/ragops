A maioria das falhas em gráficos de conhecimento (RAG) provém de duas fontes principais:
limitações do modelo e dados fracos . Embora os modelos de IA continuem a melhorar, os
problemas com os dados são frequentemente mais complexos. Muitas falhas ocorrem
quando faltam conexões nos dados , como a criação de links falsos , contexto limitado ,
entre muitos outros. Os gráficos de conhecimento podem ajudar a resolver esses
problemas, conectando e organizando os dados explicitamente. Alguns dos problemas mais
comuns em gráficos de conhecimento incluem…
14 falhas comuns do RAG (Criado por)
Ambiguidade de Entidade: Quando o sistema se confunde com termos que parecem
idênticos, mas se refere a coisas completamente diferentes, dependendo do contexto.
Desconexão Multi-Hop: Quando uma resposta exige conectar os pontos entre fatos
separados que estão dispersos em diferentes arquivos ou documentos.
Informação conflitante: ocorre quando o banco de dados contém fatos contraditórios,
como versões antigas versus novas, e o modelo não consegue determinar qual é a
versão verdadeira atual.
Falsas ligações: Quando o sistema assume incorretamente que duas coisas estão
relacionadas apenas porque aparecem próximas uma da outra no texto, levando a
conclusões falsas.
Regras Ocultas Ausentes: Quando uma descrição geral em um local ignora uma
restrição, exceção ou regra crítica que está oculta em um documento completamente
diferente.
Evidências dispersas: ocorrem quando o sistema não consegue fornecer uma lista
completa de detalhes porque os atributos necessários estão fragmentados em muitas
fontes diferentes.
Falha de agregação: ocorre quando o modelo tem dificuldade em contar ou resumir
itens com precisão porque os dados estão dispersos ou usam nomes
predominantemente diferentes para a mesma coisa.

Why We Are Choosing Smaller Models for RAG
Before we look at the failures, we need to set up our test environment. Our goal is not to show
the power of a huge modern model. Instead, we will use a smaller model TinyLlama 1.1B 
Chat v1.0 and a lightweight embedding model all MiniLM L6 v2.
Large models like GPT 5 can sometimes hide problems in a weak RAG pipeline because
their built in knowledge fills in the gaps.
A smaller model cannot do this. It forces the weaknesses of RAG to appear clearly (because of
limited knowledge), which makes the value of using a Knowledge Graph much easier to see.

Multi-hop vs RAG Ingênuo (Criado por)
Por exemplo, para descobrir a moeda da cidade natal do líder do projeto, o sistema precisa se
conectar Project -> Person -> City -> Currency.
1. A busca vetorial padrão é fundamentalmente um processo de "um salto" . Ela é
excepcionalmente boa em encontrar documentos semanticamente semelhantes à
consulta original. Quase certamente encontrará o documento que liga o termo Projectà
consulta Person.
2. No entanto, é provável que não consiga recuperar o documento que liga o projeto
Personao contexto City, porque o conceito de cidade é semanticamente distante da
consulta original focada no projeto. A cadeia lógica é quebrada antes mesmo de se
formar.
Primeiro, vamos criar esse conjunto de dados problemático. Ele conterá os três documentos
críticos que formam a cadeia lógica, além de vários documentos "distratores" para enganar
uma busca vetorial simples.

Processo KG (Criado por)
1. Eles transformam o conhecimento implícito disperso nos textos em uma rede explícita e
estruturada de relações.
2. Isso nos permite questionar as conexões entre os fatos, e não apenas os fatos em si.
3. A solução é transformar nosso texto não estruturado em um Grafo de Conhecimento
estruturado.
Usaremos nosso LLM para atuar como engenheiros de dados, extraindo entidades e seus
relacionamentos (triplas) de cada documento. Em uma solução de nível de produção,
existem muitas outras maneiras de criar grafos de conhecimento, como por meio de
reconhecimento de entidades, NER, etc.
def extract_triplets_with_llm(text):
     """
    Utiliza o LLM para analisar um trecho de texto e extrair um único trio de 
relações estruturadas.
    Usamos instruções com poucos exemplos (a linha 'Exemplo:') para guiar o 
LLM ao formato de saída correto.
    """
    extraction_prompt = f """<|system|>
    Você é um Engenheiro de Grafos de Conhecimento. Extraia a relação mais 
importante da frase.
    Formato: Sujeito | Relação | Objeto
    Exemplo: "A Apple Inc. foi fundada por Steve Jobs." -> Apple Inc. | 
fundada_por | Steve Jobs
    <|usuário|>
    Frase: {texto}
    <|assistente|>"""
    
    # 1. Execute o LLM com nosso prompt de extração especializado.
    raw_out = llm.invoke(extraction_prompt)
    # 2. Analisamos a string de saída do LLM para obter apenas a resposta do 
assistente. 
    cleaned_out = raw_out.split( "<|assistant|>" )[- 1 ].strip()
    
    # 3. Adicionamos uma lógica de análise sintática robusta para lidar com 
possíveis erros de formatação do LLM.
    partes = []
    se  "|"  em cleaned_out:
        partes = [p.strip() para p em cleaned_out.split( "|" )]
        
    retornar limpo, peças
A extract_triplets_with_llmfunção é nossa abordagem principal de extração. É um
pipeline simples de "extração baseada em esquema" . Ao fornecer uma instrução de
formato muito específica ( Subject | Relation | Object) e um exemplo, estamos
forçando o LLM a agir não como um chatbot, mas como um analisador de dados
estruturados. Este é o passo fundamental para converter nosso texto plano em um grafo.

Multi-hop (Criado por)
Para cada documento, identificou corretamente o sujeito, o objeto e a relação entre eles.
Fundamentalmente, agora temos um networkxgrafo onde o Projeto Quimera está
explicitamente conectado ao Dr. Elias Thorne, que por sua vez está conectado à Cidade de
Valoria , e assim por diante.
Agora podemos responder à consulta não com uma busca semântica frágil, mas com uma
travessia robusta e lógica de grafos.
def get_graph_context(graph, query_str, hops=3):
    """
    Replaces vector search with a multi-step graph traversal algorithm.
    """
    # Step 1: Entity Resolution. We find the starting node in our graph that 
matches an entity in the user's query.
    # In a production system, this would use a more advanced technique like 
vector similarity or fuzzy matching.
    start_node = None
    for node in graph.nodes():
        if node in query_str and len(node) > 4: 
            start_node = node
            break
    
    print(f"Query Entity Detected: '{start_node}'")
    if not start_node: return "No relevant entity found in the knowledge graph 
for this query."
    
    # Step 2: Graph Traversal. We perform a Breadth-First Search (BFS) from 
our starting node to find all connected facts up to 'hops' away.
    print("\nTraversing Graph Paths...")
    path_sentences = []
    # nx.bfs_edges finds all connected edges in a breadth-first manner.
    edges = list(nx.bfs_edges(graph, source=start_node, depth_limit=hops))
    
    # We convert the found graph path back into natural language sentences for 
the final generation step.
    for u, v in edges:
        rel = graph[u][v]['relation']
        sentence = f"{u} has a relation '{rel}' with {v}." 
        path_sentences.append(sentence)
    
    return " ".join(path_sentences)
Inget_graph_context function we are replacing the one-hop vector search with a multi-hop
graph algorithm.
1. First, it identifies a starting entity in the graph (Project Chimera).
2. Then, it uses a standard graph traversal algorithm (nx.bfs_edges) to programmatically
follow the explicit connections we created, hopping from Project Chimera to Dr. 
Elias Thorne and so on.
Let’s run our new, graph-powered pipeline.

Se um usuário perguntar "X é compatível com Y?", um sistema RAG básico buscará
documentos sobre X e Y. Se esses documentos contiverem frases de marketing com tom

Ela se concentra nas palavras positivas e pode afirmar erroneamente que a combinação é
segura, mesmo que exista uma importante regra de segurança que a proíba.

def  verificar_segurança ( chem_a, chem_b ):
     """
    Replaces LLM synthesis with a deterministic logic engine that queries the 
graph.
    """
    print(f"Checking Compatibility: {chem_a}  VS  {chem_b}")
    
    # Step 1: Graph Traversal. For each chemical, we find its class and its 
inherited properties.
    def get_properties(node):
        props = set()
        # Find the class of the chemical (e.g., Titan-X -> Peroxide)
        classes = list(kg.successors(node))
        for cls in classes:
            # Find the properties of that class (e.g., Peroxide -> Oxidizer)
            class_props = list(kg.successors(cls))
            props.update(class_props)
        return classes, props
    class_a, props_a = get_properties(chem_a)
    class_b, props_b = get_properties(chem_b)
    
    print(f"\nAnalyzing '{chem_a}':\n  - Is instance of: {class_a}\n  - 
Inherited Property: {props_a}")
    print(f"\nAnalyzing '{chem_b}':\n  - Is instance of: {class_b}\n  - 
Inherited Property: {props_b}")
    
    # Step 2: Deterministic Rule Check. We apply our safety rule 
programmatically.
    print("\nCONFLICT CHECK:")
    print("  Checking Rule: (Oxidizer + Flammable)...")
    if ('Oxidizer' in props_a and 'Flammable' in props_b) or \
       ('Flammable' in props_a and 'Oxidizer' in props_b):
        print("  !!! VIOLATION FOUND !!!")
        print("  Rule: Oxidizers cannot be stored with Flammable material.")
        return "UNSAFE"
    
    return "SAFE"
The check_safety function replaces the probabilistic reasoning of the LLM with a two-step
logical process.
1. First, it performs a graph traversal to determine the inherited properties of each chemical.
2. Second, it applies a hardcoded safety rule. This process is deterministic, verifiable, and
guaranteed to be correct based on the information in the graph.

def  resolve_and_query_graph ( query ):
     """
    Substitui a busca vetorial por um processo de duas etapas: resolução de 
intenção e recuperação de grafo filtrado.
    """
    # Step 1: Resolve User Intent. In a production system, this would be a 
classifier or another LLM call.
    # For this demo, we'll simulate that the system has correctly identified a 
'Financial' intent.
    def resolve_query_intent(q):
        if "perform" in q or "revenue" in q:
            return "Company"
        return "Unknown"
    print(f"User Query: '{query}'")
    intent_type = resolve_query_intent(query)
    print(f"Detected Intent Type: {intent_type.upper()}")
    print("\nGraph Search (Filtered by Intent):")
    relevant_facts = []
    
    # Step 2: Filtered Traversal. We iterate through our unique graph nodes.
    for node, attrs in kg.nodes(data=True):
        node_type = attrs.get('type', 'Unknown')
        
        # This is the core disambiguation logic. We only consider nodes that 
match our resolved intent.
        if intent_type == node_type: 
            print(f"  - Found node '{node}' -> MATCHES INTENT! Retrieving 
facts.")
            # We get all the facts connected to this specific, disambiguated 
node.
            neighbors = list(kg.successors(node))
            for n in neighbors:
                rel = kg[node][n]['relation']
                relevant_facts.append(f"{node} {rel} {n}.")
        elif "Jaguar" in node:
            # We explicitly log the nodes that were ignored.
            print(f"  - Found node '{node}' -> IGNORED (Type mismatch).")
            
    filtered_context = " ".join(relevant_facts)
    print(f"\nFiltered Context: {filtered_context}")
    return filtered_context
It replaces the single, ambiguous vector search with a precise, two-step logical process.
1. First, it determines the user intent (in this case Company).
2. Second, it performs a filtered search on the graph, completely ignoring any nodes that
do not match this intent. This guarantees that the context provided to the final LLM is not
only relevant but also free from the ambiguity that plagued the naive RAG system.

Handling Conflicting Versions
We have seen how Knowledge Graphs can resolve ambiguity. Now we address a failure mode
that plagues any real-world, evolving knowledge base: Contradictory Information.
Políticas corporativas, especificações técnicas e documentos legais não são estáticos, sendo
constantemente atualizados. Um banco de dados vetorial, no entanto, não possui um conceito
inerente de tempo. Documentos antigos (por exemplo, uma política de 2021) frequentemente
coexistem com documentos novos (por exemplo, uma política de 2024) no índice.

Falha conflitante (Criada por)
Quando um usuário faz uma pergunta simples como "Qual é a política?", uma busca
vetorial ingênua em um sistema RAG geralmente recupera todas as versões relevantes
porque elas são semanticamente idênticas, diferindo apenas por uma data ou um número.
Diante desse contexto contraditório, o LLM (Licensed Liability Management) se mostra
incapaz de distinguir a verdade atual dos fatos históricos . Frequentemente, ele opta por
uma resposta evasiva, apresentando todas as opções conflitantes ao usuário ou, pior ainda,
criando uma mistura confusa das políticas antigas e novas.

documentos descrevendo a mesma política em três momentos diferentes.

Política Temporal (Criada por)
O resultado mostra que nosso pipeline criou com sucesso um único Remote Work Policynó
em nosso grafo. Anexado a esse nó está um historyatributo contendo uma lista de todos os
três fatos conflitantes, mas, crucialmente, cada fato agora está emparelhado com um objeto
legível por máquina e classificável datetime. Estruturamos a natureza temporal dos dados.
Com esse grafo temporal construído, agora podemos responder à consulta com uma
operação de classificação determinística, contornando completamente o raciocínio falho do
LLM.
def  resolve_latest_truth ( topic_query ):
     """
    Substitui a síntese LLM por um processo determinístico de classificação de 
fatos por carimbo de data/hora.
    """ 
    # Etapa 1: Encontre o nó de tópico relevante no grafo. 
    target_node = None 
    for node in kg.nodes():
         if  "Remote Work"  in node or  "work from home"  in topic_query:
            nó_alvo = nó
            quebrar
    
    print ( f"Consulta: ' {topic_query} '" )
     print ( f"Tópico identificado: ' {target_node} '" )
    
    Se  não houver target_node:
         retorne  "Nenhuma política encontrada neste tópico".
    
    # Etapa 2: Recupere todo o histórico de fatos para esse tópico a partir do 
atributo do nó. 
    history = kg.nodes[target_node][ 'history' ]
     print ( f"\nEncontrados { len (history)} registros conflitantes. 
Resolvendo por data..." )
    
    # Step 3: Sort the history list in descending order based on the 'date' 
key.
    # This is the core of the temporal resolution logic.
    sorted_history = sorted(history, key=lambda x: x['date'], reverse=True)
    
    for idx, record in enumerate(sorted_history):
        status = "(Latest)" if idx == 0 else ""
        print(f"  {idx+1}. Value: {record['value']} | Date: 
{record['date'].date()} {status}")
        
    # Step 4: The latest truth is simply the first item in the sorted list.
    latest = sorted_history[0]
    print(f"\nSelected Truth: {latest['value']} (Source: 
{latest['source']}...)")
    
    return latest
The resolve_latest_truth function replaces the LLM unreliable attempt at reasoning with a
simple, robust, and programmatic sort. It retrieves all historical versions of the policy from the
graph node, sorts them using Python's built-in sorted() function, and definitively selects the
first item as the current truth. This is not a guess; it's a deterministic algorithm.

Falhas em links falsos (Criado por)
Assim, quando um usuário pergunta sobre o relacionamento entre as empresas, um sistema
RAG básico extrai esse parágrafo. O LLM então compara as empresas lado a lado e
frequentemente inventa uma conexão real, como uma parceria ou aquisição, mesmo que ela
não exista.
Considera erroneamente a simples coocorrência como prova de uma relação.
Primeiro, vamos criar nosso conjunto de dados adversário. Ele conterá um documento que
declara uma relação verificada e um documento "armadilha" onde nossas entidades-alvo são
meramente mencionadas na mesma frase.

Processo KG (Criado por)
A solução é construir um Grafo de Conhecimento usando um processo de extração rigoroso,
orientado por esquema. Definiremos uma lista de relações ALLOWED_RELATIONSe
adicionaremos triplas ao nosso grafo somente se a relação extraída estiver nessa lista.
def strict_extract(text):
     """
    Utiliza o LLM para extrair uma relação comercial, que é então filtrada por 
um esquema predefinido.
    """ 
    extraction_prompt = f """<|system|>
    Extraia a relação comercial entre as empresas mencionadas.
    Formato: Entidade A | RELAÇÃO | Entidade B
    Use relações concisas com um único verbo, como: FORNECE, FAZ PARCERIA COM, 
DESENVOLVE, DISCUSSOU COM, COMPARECEU.
    <|usuário|>
    Texto: {texto}
    <|assistente|>"""
    
    # 1. Execute o LLM para extrair qualquer relação potencial.
    raw_out = llm.invoke(extraction_prompt)
    
    # 2. Analise a saída. 
    cleaned_out = raw_out.split( "<|assistant|>" )[- 1 ].strip()
    
    # 3. Lidar com possíveis erros de formatação.
    partes = []
    se  "|"  em cleaned_out:
        partes = [p.strip() para p em cleaned_out.split( "|" )]
        
    peças de devolução
A strict_extractfunção é o nosso pipeline de extração inicial. Ela instrui o LLM a identificar
uma relação no texto. No entanto, a parte real acontece depois que essa função é chamada,
onde aplicaremos nosso filtro programático.
Agora, vamos percorrer todos os nossos documentos e usar essa função, combinada com
nosso ALLOWED_RELATIONSfiltro, para construir nosso grafo limpo e verificado.

Evidências Esparsas (Criadas por)
O sistema RAG padrão utiliza um top_kparâmetro (geralmente de 3 a 5) para limitar o
número de documentos enviados ao LLM.
Se um usuário solicitar uma lista completa, como " Liste todos os recursos de segurança do
Modelo X" , e esses recursos forem mencionados um a um em 10 páginas diferentes de um
manual, o recuperador vetorial, limitado por top_k, buscará apenas um pequeno subconjunto
dessas páginas. O LLM, vendo apenas uma lista parcial, apresentará com confiança (e
incorretamente) uma resposta incompleta.
Primeiro, vamos criar o conjunto de dados que demonstrará esse problema. Ele conterá cinco
documentos distintos, cada um descrevendo uma característica única do "Modelo-X", além
de um documento de distração.

Relação sujeito/objeto (Criada por)
Quando um sistema RAG ingênuo recupera documentos que contêm essas informações,

def analyze_ownership_from_graph(entity_name):
    """
    Replaces LLM synthesis with a deterministic query of the graph's directed 
edges.
    """
    print(f"Target Node: '{entity_name}'")
    
    if entity_name not in kg:
        return "Entity not found in the knowledge graph."
    
    # To find who owns our target, we look for its PREDECESSORS (nodes with an 
edge pointing TO it).
    parents = list(kg.predecessors(entity_name))
    print(f"\nAnalyzing Upstream (Predecessors) -> i.e., Who OWNS 
'{entity_name}'?")
    if parents:
        for p in parents:
            print(f"   FOUND: {p}")
    else:
        print("   No owners found (The company is independent).")
        
    # To find who our target owns, we look for its SUCCESSORS (nodes with an 
edge pointing FROM it).
    children = list(kg.successors(entity_name))
    print(f"\nAnalyzing Downstream (Successors) -> i.e., Who does 
'{entity_name}' OWN?")
    if children:
        for c in children:
            print(f"   FOUND: {c}")
    else:
        print("   No subsidiaries found.")
            
    # We can now construct a precise, unambiguous answer programmatically.
    answer = f"{entity_name} is owned by {', '.join(parents) if parents else 
'no one'}."
    if children:
        answer += f" However, {entity_name} itself owns {', 
'.join(children)}."
    return answer
The kg.predecessors(target) function deterministically finds all nodes that have target as
their child, giving us the owners. Conversely, kg.successors(target) finds all children.
These operations are unambiguous and respect the immutable direction of the edges we
created.

O sistema KG-RAG adiciona direção aos dados à medida que são recebidos. Ele usa um
Modelo de Linguagem Lógica (LLM) para rotular as funções em uma frase e as armazena em
um grafo direcionado. Isso transforma um problema de compreensão de linguagem em uma
estrutura lógica e clara.
Para questões sobre relações direcionais, como propriedade, causa e efeito ou hierarquia,
um grafo direcionado é crucial para obter respostas precisas.

Classificação de Relevância (Criada por)
Isso cria um modo de falha crítico no RAG. Um recuperador vetorial padrão classifica os
documentos encontrados por relevância semântica , não por carimbo de data/hora .
1. Se um usuário perguntar: "O que causou a falha do servidor?" , o sistema
inevitavelmente classificará a entrada de log que contém a frase "Falha do servidor"
como a mais alta, porque é a correspondência semântica mais adequada.
2. Isso colocará a entrada de registro sobre o "e-mail de phishing" inicial , que é
semanticamente distante, muito mais abaixo no contexto recuperado.
3. O LLM então lê o efeito antes de ler a causa, e frequentemente falha em reconstruir a
linha do tempo corretamente, muitas vezes alucinando que um evento posterior
desencadeou um anterior.

def  find_trigger_event ( target_keyword ):
     """
    Substitui a síntese LLM por uma travessia determinística do grafo da série 
temporal.
    """ 
    print ( "\n1. Localizando o Evento Alvo no Grafo..." )
     # Etapa 1: Encontre o nó em nosso grafo que corresponde ao evento na 
consulta. 
    target_node = None 
    for node in kg.nodes():
         if target_keyword in node:
            nó_alvo = nó
            quebrar
            
    Se  não target_node:
         retorne  "Evento alvo não encontrado nos logs." 
    print ( f" Alvo identificado: ' {target_node} '" )
    
    # Etapa 2: Realizar uma busca reversa. Encontramos os PREDECESSORES do 
nosso nó alvo. 
    # Em nossa cadeia de séries temporais simples, cada evento terá no máximo 
um predecessor. 
    print ( "\n2. Realizando Busca Reversa (Encontrando Predecessores)..." )
    predecessores = lista (kg.predecessores(nó_alvo))
    
    Se  não houver predecessores:
         retorne  "Este parece ser o primeiro evento na sequência (a causa 
raiz)."
    
    # O gatilho imediato é o único predecessor em nossa cadeia. 
    trigger = predecessors[ 0 ]
     print ( f" Predecessor encontrado: ' {trigger} '" )

Problema de dados aninhados (Criado por)
Muitas vezes, essa informação está fragmentada e dispersa em vários documentos. Um
documento pode afirmar "O Sistema A contém o Módulo B", enquanto um documento
completamente separado afirma "O Módulo B contém a Peça C".
Quando um usuário faz uma pergunta sobre a parte mais profunda dessa hierarquia, como
"Rastreie a linhagem completa do sistema para a Parte C", um sistema RAG ingênuo
provavelmente falhará.
Sua busca vetorial, focada nas palavras-chave “Parte C”, recuperará o documento que a
vincula ao “Módulo B”. No entanto, frequentemente falhará em recuperar o documento que
vincula o “Módulo B” ao “Sistema A”, porque a entidade “Sistema A” é semanticamente
distante da “Parte C” e pode não aparecer nos primeiros kresultados.
The LLM is then presented with only a fragment of the hierarchy and provides an incomplete
answer, losing the "big picture" context.
Our sythetic dataset contains three separate documents, each defining one level of a nested
product hierarchy.

def  check_ddi ( a, b ):
     """
    Substitui a síntese LLM por uma consulta gráfica determinística para 
encontrar vizinhos comuns e aplicar uma regra lógica.
    """ 
    se a não estiver  em kg ou b não estiver  em kg:
         retorne  "Um ou ambos os medicamentos não foram encontrados no grafo 
de conhecimento."
    
    # Passo 1: Encontrar vizinhos comuns. Obtemos o conjunto de todos os 
sucessores (vizinhos) para cada droga. 
    targets_a = set (kg.successors(a))
    targets_b = set(kg.successors(b))
    # We find the intersection of these two sets.
    intersection = targets_a.intersection(targets_b)
    
    print(f"\n1. Common Neighbors Analysis:")
    print(f"   Neighbors of {a}: {targets_a}")
    print(f"   Neighbors of {b}: {targets_b}")
    print(f"   INTERSECTION: {intersection}")
    
    print(f"\n2. Logic Check on Intersections:")
    risk_found = False
    explanation = ""
    
    # Step 2: Apply a Logic Rule to each common neighbor.
    for target in intersection:
        print(f"   Checking common neighbor '{target}'...")
        rel_a = kg[a][target]['relation']
        rel_b = kg[b][target]['relation']
        
        print(f"     Found Path: {a} -> {rel_a} -> {target}")
        print(f"     Found Path: {b} -> {rel_b} -> {target}")
        
        # This is our hardcoded pharmacological rule.
        if (rel_a == "INHIBITS" and rel_b == "METABOLIZED_BY") or \
           (rel_a == "METABOLIZED_BY" and rel_b == "INHIBITS"):
            print("     -> Logic Rule Triggered: DDI ALERT! An Inhibitor and a 
Substrate share the same enzyme.")
            risk_found = True
            explanation = f"CRITICAL WARNING: Interaction detected. {a} 
{rel_a} {target}, while {b} is {rel_b} {target}. This combination can lead to 
toxic levels of {b}."
        else:
            print("     -> Logic Rule Not Triggered: Harmless co-occurrence.")
            
    if not risk_found:
        return "No critical interaction pattern found between the two drugs."
    return explanation
The check_ddi function is the core of our KG solution. It replaces the probabilistic synthesis of
the LLM with a two-step logical algorithm.
1. First, it performs an intersection query on the graph to find any and all biological
targets that the two drugs have in common.
2. Second, for each of these common neighbors, it applies a deterministic logic rule. This
process is not a guess; it's a programmatic search for a specific, dangerous interaction
pattern.

def  analyze_network ():
     """
    Substitui a síntese LLM por uma análise matemática determinística da 
topologia do grafo.
    """ 
    print ( "\nExecutando Análise Matemática na Topologia de Grafos..." )
    
    # A centralidade de intermediação é uma métrica da teoria dos grafos que 
mede a frequência com que um nó 
    # aparece no caminho mais curto entre quaisquer outros dois nós. Uma 
pontuação alta significa que o nó é uma "ponte" crítica. 
    centralidade = nx.centralidade_de_intermediação(kg, peso= 'peso' )
    
    # Ordenamos os nós por sua pontuação de centralidade em ordem decrescente. 
    sorted_nodes = sorted (centrality.items(), key= lambda x: x[ 1 ], reverse= 
True )
    
    print("\nCentrality Scores (0.0 to 1.0):")
    for node, score in sorted_nodes:
        print(f"   {node}: {score:.2f}")
        
    # The node with the highest score is our bottleneck.
    top_person = sorted_nodes[0][0]
    top_score = sorted_nodes[0][1]
    
    print("\nTopological Insight:")
    print(f"- Alice (the PM) only connects to {top_person}.")
    print(f"- The Dev Team only connects to {top_person}.")
    print(f"- Therefore, if {top_person} is removed from the network, the 
graph would shatter into multiple disconnected components.")
    
    # We construct a definitive answer based on the mathematical result.
    return f"Network analysis identifies {top_person} as the critical 
bottleneck with a Betweenness Centrality score of {top_score:.2f}. This person 
is the sole bridge connecting the management team (Alice, Bob) to the 
operational teams (Dev, QA)."
The analyze_network function is the core of our KG solution. It completely removes the LLM
from the final reasoning process. Instead, it uses the powerful nx.betweenness_centrality
function from the networkx library.
It is a mathematical calculation that proves which node is the most critical bridge in the
network. The function then uses this mathematical proof to construct a definitive, evidence-
based answer.

Conclusion
So we have looked into the specific limitations of Naive RAG and demonstrated how a
Knowledge Graph architecture transforms these failures into solvable, deterministic problems.
Here is what can be done next to take this from a prototype to production:
1. Scale to Production Databases: While NetworkX is perfect for tutorials, real-world
applications require dedicated graph databases like Neo4j, ArangoDB, or AWS Neptune
to handle millions of nodes efficiently.
2. Implement Hybrid RAG: You don’t have to choose one over the other. The most robust
systems use Hybrid RAG, combining the structural precision of Knowledge Graphs with
the semantic flexibility of Vector Search.
3. Automatize a construção de grafos: escrever prompts de extração personalizados
consome muito tempo. Explore frameworks como o GraphRAG da Microsoft ou o
LlamaIndex para automatizar o fluxo de trabalho de conversão de texto não estruturado
em entidades e relacionamentos.
